# 计算机网络深度剖析：TCP 协议栈、HTTP/2/3、TLS 1.3

> 面向高级开发者的网络协议内核原理

## 一、TCP 协议

### 1.1 TCP 状态机

```c
// TCP 的 11 种状态
// LISTEN：等待连接
// SYN-SENT：发送 SYN 后
// SYN-RECEIVED：收到 SYN 后
// ESTABLISHED：连接建立
// FIN-WAIT-1：发送 FIN 后
// FIN-WAIT-2：收到 ACK 后
// CLOSE-WAIT：收到 FIN 后
// CLOSING：双方同时关闭
// LAST-ACK：等待最后一个 ACK
// TIME-WAIT：等待 2MSL
// CLOSED：关闭
```

### 1.2 拥塞控制算法

```c
// TCP 拥塞控制算法演进
// 1. Tahoe：慢启动 + 拥塞避免 + 快重传
// 2. Reno：Tahoe + 快恢复
// 3. NewReno：改进快恢复（处理多个丢包）
// 4. BIC：二分搜索拥塞窗口
// 5. CUBIC：三次函数拥塞窗口（Linux 默认，5.8+）

// BBR（Bottleneck Bandwidth and Round-trip）
// Google 提出的基于带宽和延迟的拥塞控制
// 不依赖丢包检测
// 适合高带宽高延迟网络
```

### 1.3 TIME_WAIT 详解

```c
// TIME_WAIT 状态（主动关闭方）
// 持续 2MSL（Maximum Segment Lifetime）
// MSL 通常为 30 秒，所以 TIME_WAIT 为 60 秒

// 目的
// 1. 确保最后一个 ACK 到达对方
// 2. 防止旧连接报文干扰新连接

// 问题
// 大量 TIME_WAIT 连接消耗资源
// 解决方案：
// 1. 调整 MSL 时间
// 2. 启用 SO_REUSEADDR
// 3. 使用 TCP 快速打开（TFO）
```

### 2.1 二进制分帧

```c
// HTTP/2 使用二进制帧格式
// 帧类型
// 1. DATA：数据帧
// 2. HEADERS：头帧

// 10. CONTINUATION：续传帧

// Length (24 bits) + Type (8 bits) + Flags (8 bits) + 
// Stream Identifier (32 bits) + Frame Payload
```

// 4. 解决 HTTP/1.1 的队头阻塞问题

// 流量控制
// 基于流级别的流量控制
// 初始窗口大小 64KB
// 通过 SETTINGS 帧调整

## 三、HTTP/3

### 3.1 QUIC 协议

```c
// HTTP/3 基于 QUIC（Quick UDP Internet Connections）
// QUIC 是 Google 开发的传输层协议

// QUIC 优势
// 1. 0-RTT 连接建立
// 2. 内置 TLS 1.3
// 3. 连接迁移（IP 变化不断连）
// 4. 无队头阻塞（独立流）
// 5. 前向纠错

// QUIC 连接建立
// 1. 首次连接：1-RTT（TLS 1.3 握手）
// 2. 再次连接：0-RTT（使用缓存参数）
```

## 四、TLS 1.3

### 4.1 握手流程

```c
// TLS 1.3 握手（1-RTT）
Client → ClientHello + KeyShare → Server
Client ← ServerHello + KeyShare + Certificate + Finished ← Server
Client → ClientFinished → Server
Client ↔ Application Data ↔ Server

// 0-RTT 模式
// 使用 PSK（Pre-Shared Key）
// 首次连接后缓存会话参数
// 再次连接时直接发送数据
```

### 4.2 加密套件

```c
// TLS 1.3 只支持 5 个加密套件
// TLS_AES_128_GCM_SHA256
// TLS_AES_256_GCM_SHA384
// TLS_CHACHA20_POLY1305_SHA256
// TLS_AES_128_GCM_SHA256
// TLS_AES_128_CCM_SHA256

// 移除的加密套件
// RSA 密钥交换（不支持前向保密）
// CBC 模式（易受攻击）
// RC4（已破解）
// 3DES（慢且不安全）
```

---

> **总结**：TCP 通过拥塞控制算法（CUBIC、BBR）优化网络性能，HTTP/2 通过二进制分帧和多路复用解决队头阻塞，HTTP/3 通过 QUIC 协议实现 0-RTT 连接和连接迁移，TLS 1.3 通过简化握手流程提升安全性。理解这些协议的原理是排查网络问题和优化应用性能的关键。

## 面试题精选

**Q1: TCP 三次握手为什么是三次，不是两次？**

A: 防止已失效的连接请求到达服务器。两次握手无法确认客户端的接收能力，可能导致服务器一直等待。

**Q2: TIME_WAIT 状态的作用是什么？**

A: 1）确保最后一个 ACK 到达对方；2）防止旧连接报文干扰新连接。持续 2MSL（约 60 秒）。

**Q3: HTTP/1.1、HTTP/2、HTTP/3 的主要区别？**

A: HTTP/1.1 长连接+管道化，有队头阻塞；HTTP/2 多路复用+二进制分帧+头部压缩，TCP 层仍有队头阻塞；HTTP/3 基于 QUIC（UDP），0-RTT，无队头阻塞。

**Q4: HTTPS 的握手流程是怎样的？**

A: 客户端请求 → 服务器返回证书（公钥+CA 签名）→ 客户端验证证书 → 生成对称密钥用公钥加密发送 → 服务器用私钥解密 → 对称加密通信。

**Q5: 在浏览器输入 URL 后发生了什么？**

A: DNS 解析 → TCP 三次握手 → TLS 握手（HTTPS）→ 发送 HTTP 请求 → 服务器处理返回 → 浏览器解析渲染 → TCP 四次挥手。
