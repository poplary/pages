```c
// 1. 首次适应（First Fit）
// 从头扫描，找到第一个合适的块
// 速度快，但可能产生碎片

// 2. 最佳适应（Best Fit）
// 找到最合适的块
// 可能会产生小碎片

// 3. 最差适应（Worst Fit）
// 找到最大的块
// 减少小碎片
```

## 二、分布式系统

### 2.1 Raft 协议

```c
// Raft 的核心组件
// 1. Leader：处理所有客户端请求
// 2. Follower：被动响应
// 3. Candidate：竞选 Leader

// 任期（Term）
// 每个 Leader 对应一个任期
// 选举超时触发新任期

// 日志复制
// 1. Leader 接收客户端请求
// 2. 追加到本地日志
// 3. 并行发送到 Follower
// 4. 多数派确认后提交
// 5. 通知 Follower 提交
```

### 2.2 分布式一致性

```c
// 一致性级别
// 1. 强一致性：所有节点看到相同数据
// 2. 最终一致性：最终所有节点一致
// 3. 因果一致性：有因果关系的操作有序
// 4. 会话一致性：同一会话内一致
// 5. 单调读：读到的数据不会回退
```

## 面试题精选

**Q1: CAP 定理是什么？如何选择？**

A: 一致性（C）、可用性（A）、分区容错性（P）三者只能选其二。CP（ZooKeeper/Etcd）、AP（Eureka/Cassandra）、CA（单机数据库）。

**Q2: Raft 协议的 Leader 选举过程？**

A: Follower 超时未收到心跳 → 转为 Candidate → 自增任期 → 请求投票 → 获得多数票 → 成为 Leader → 定期发送心跳维持地位。

**Q3: 如何设计一个秒杀系统？**

A: 前端限流（按钮置灰+验证码）+ 网关限流（令牌桶）+ Redis 预减库存 + MQ 异步下单 + 数据库乐观锁。

**Q4: 分布式事务有哪些解决方案？**

A: 2PC（两阶段提交，强一致）、TCC（Try-Confirm-Cancel，业务补偿）、本地消息表（最终一致）、事务消息（RocketMQ）、Saga（补偿操作）。
